# 资源位后台版本发布 SOP 流程

**文档版本**：V1.1
**适用范围**：资源位后台配置系统的版本发布管理
**最后更新**：2026-05-24

---

## 一、概述

### 1.1 目的

规范资源位后台配置系统的版本发布流程，确保每次版本变更可追溯、可验证、可回滚，降低发布风险。

### 1.2 适用范围

本 SOP 覆盖资源位后台所有涉及客户端渲染变更的发布场景，包括但不限于：

| 场景 | 示例 |
|------|------|
| 新增资源位类型 | 新增"AI交易套件"入口、新增导航菜单项 |
| 现有资源位字段变更 | 弹窗新增"奖励阶段"子表、跑马灯新增"文本颜色" |
| 运营控制项升级 | 新增"灰度比例"字段、AB测试分组扩展 |
| 展示逻辑调整 | 优先级计算规则变更、频控策略更新 |
| 异常处理策略变更 | 降级兜底方案修改 |
| 多语言配置新增/变更 | 新增语言包、翻译修正 |
| **事件驱动配置上线**（V1.1 新增） | 新币上线/Launchpad/交易大赛等事件自动触发资源位配置，运营审核后上线 |

### 1.3 版本号规范

```
主版本.次版本.修订号

主版本（Major）：资源位架构级变更（如新增一类资源位体系）
次版本（Minor）：功能新增/字段扩展
修订号（Patch）：Bug 修复、文案调整、配置项微调
```

| 版本类型 | 示例 | 发布节奏 |
|----------|------|----------|
| 主版本 | V2.0 新增"交易策略推广位"体系 | 重大架构变更时发布 |
| 次版本 | V1.8 新增"AI交易套件"导航菜单配置 | 按需发布（约每2周） |
| 修订号 | V1.7.1 修复弹窗频控计算逻辑 | 随时发布（hotfix） |

### 1.4 角色与职责

| 角色 | 职责 |
|------|------|
| **产品运营** | 提交发布需求，确认业务验收 |
| **配置管理员** | 起草配置变更内容，执行发布操作 |
| **技术负责人** | 代码/配置 Review，审批技术方案 |
| **QA测试** | 测试环境验证，回归测试 |
| **发布负责人** | 最终审批，执行/监督上线，负责回滚决策 |

## 二、发布类型与流程矩阵

| 发布类型 | 触发条件 | 审批层级 | 测试要求 | 灰度要求 | 全量时间 |
|----------|----------|----------|----------|----------|----------|
| **常规版本发布** | 按计划的功能迭代 | 配置管理员 → 技术负责人 → 发布负责人 | 完整回归测试 | 是（A渠道先行） | 灰度验证24h后全量 |
| **紧急热修复** | 线上 Bug、配置错误 | 技术负责人 → 发布负责人（口头确认即可） | 最小化验证 | 可选 | 视紧急程度决定 |
| **配置参数调整** | 字段值修改（不涉及结构变更） | 配置管理员自审 | 预览验证 | 否 | 即时生效 |
| **活动配置上线** | 营销活动相关配置 | 运营 + 配置管理员双审 | 预览验证 | 否 | 定时上线 |
| **事件驱动配置上线**（V1.1 新增） | 新币上线/Launchpad/交易大赛自动触发 | 运营审核确认（单审） | 预览验证 | 否 | 与事件同步上线 |

## 三、常规版本发布流程

### 3.1 流程图

```
需求评审 → 方案设计 → 配置起草 → 技术Review → 测试验证 → 灰度发布 → 数据验证 → 全量发布 → 发布总结
  (0.5d)   (0.5d)    (1d)      (0.5d)      (1d)       (1d)      (4h)      (即时)     (0.5d)
```

### 3.2 详细步骤

#### 阶段一：需求评审（Day 1 上午）

| 步骤 | 操作人 | 操作内容 | 产出物 |
|------|--------|----------|--------|
| 1.1 | 产品运营 | 提交资源位需求，明确资源位类型、目标人群、预期效果 | 需求文档/需求卡片 |
| 1.2 | 配置管理员 | 对照[资源位配置规范文档](./资源位后台配置规范文档.md)判断需求是否可行 | 可行性评估 |
| 1.3 | 配置管理员 + 技术负责人 | 评估是否需要新增字段、修改展示逻辑、调整优先级 | 技术评估意见 |

**准入标准**：
- [ ] 需求已明确资源位类型和投放目标
- [ ] 需求不与现有排期冲突
- [ ] 技术可行性已确认

#### 阶段二：方案设计（Day 1 下午）

| 步骤 | 操作人 | 操作内容 | 产出物 |
|------|--------|----------|--------|
| 2.1 | 配置管理员 | 确定配置变更范围（新增/修改/删除的字段） | 变更清单 |
| 2.2 | 配置管理员 | 设计人群定向方案（使用哪些人群包，AND/OR 逻辑） | 人群定向方案 |
| 2.3 | 配置管理员 | 规划灰度发布策略（灰度比例、灰度渠道、灰度时长） | 灰度方案 |
| 2.4 | 技术负责人 | 确认变更对客户端渲染、缓存策略的影响 | 技术影响评估 |

**灰度策略参考**：

| 资源位重要度 | 灰度比例起步 | 灰度观察时间 | 放量节奏 |
|-------------|-------------|-------------|----------|
| 高（弹窗/Banner） | 10% | 24h | 10% → 30% → 50% → 100% |
| 中（四宫图/任务位） | 20% | 12h | 20% → 50% → 100% |
| 低（跑马灯/热词） | 50% | 4h | 50% → 100% |

**产出物检查**：
- [ ] 变更清单已列明所有受影响的资源位和字段
- [ ] 人群定向方案符合[人群定向配置规范](./资源位后台配置规范文档.md#25-人群定向配置)
- [ ] 灰度方案已设定各阶段的灰度比例和观察指标

#### 阶段三：配置起草（Day 2）

| 步骤 | 操作人 | 操作内容 | 产出物 |
|------|--------|----------|--------|
| 3.1 | 配置管理员 | 在后台填写新增/修改的配置字段 | 草稿配置 |
| 3.2 | 配置管理员 | 设置 `is_active = false`（先不上线） | — |
| 3.3 | 配置管理员 | 配置排期（含定时上下线时间） | 排期配置 |
| 3.4 | 配置管理员 | 设置灰度比例初始值、关联 AB 测试分组（如有） | 运营控制配置 |
| 3.5 | 配置管理员 | 多语言文案同步配置（如有涉及） | 多语言配置 |

**字段检查清单**：

| 检查项 | 规则 | 依据 |
|--------|------|------|
| 优先级 | 不与同资源位已有配置的优先级重复 | [资源位优先级汇总](./资源位后台配置规范文档.md#四-资源位优先级汇总) |
| 排期 | 不与同资源位已有配置时间重叠 | [投放排期配置](./资源位后台配置规范文档.md#28-投放排期配置) |
| 图片尺寸 | 符合资源位的建议尺寸，偏差 ≤ 20% | 各资源位字段配置表 |
| 跳转链接 | 链接有效可访问，支持 DeepLink 格式 | 各资源位异常处理表 |
| 频控 | 频控值在合理范围内且不与其他资源位冲突 | [通用运营控制项](./资源位后台配置规范文档.md#通用运营控制项) |
| 人群定向 | AND/OR 逻辑明确，人群包非空 | [人群定向配置](./资源位后台配置规范文档.md#25-人群定向配置) |
| 上线开关 | **必须为关闭状态** | — |

**V1.1 新增自动化校验**：

| 校验项 | 触发时机 | 处理方式 |
|--------|----------|----------|
| **排期冲突检测** | 配置保存时 | 实时检测同资源位时间重叠，有冲突则**强制拦截**，提示冲突详情 |
| **人群包非空校验** | 配置保存时 | 查询人群包预估人数，为 0 时**强制拦截**，提示"人群包为空，无法保存" |
| **人群包上线前巡检** | 定时上线前 30 分钟 | 自动巡检即将上线配置的人群包，为空的**阻止上线**并通知 |
| **灰度目标人数校验** | 灰度比例调整时 | 校验目标比例下覆盖人数 ≥ 1000，不足时提示"样本量不足" |
| **NEW 标识超期预警** | 每日巡检 | NEW 标识持续超 30 天自动提醒，超 45 天自动关闭 |
| **活动排期一致性** | 配置保存时 | 资源位投放时间超出活动时间范围时**警告提示** |

#### 阶段四：技术 Review（Day 2 下午）

| 步骤 | 操作人 | 操作内容 |
|------|--------|----------|
| 4.1 | 技术负责人 | Review 所有配置字段的完整性和正确性 |
| 4.2 | 技术负责人 | 确认展示逻辑、异常处理、降级策略是否正确 |
| 4.3 | 技术负责人 | 检查多资源位之间的优先级和互斥关系 |
| 4.4 | 技术负责人 + 配置管理员 | 确认 AB 测试分组关联（如有） |

**Review 重点**：

| 资源位类型 | 重点检查项 |
|-----------|-----------|
| 弹窗类 | 频控逻辑、弹窗层级（352/353）、奖励阶段概率总和 |
| 任务类 | 完成条件校验、奖励发放方式、任务有效期 |
| 展示类 | 图片加载异常兜底、缓存策略、默认占位图 |
| PUSH 推送 | 频控规则、免打扰时段、分批投递比例、人群包非空校验、渠道降级策略 |
| 导航菜单 | 人群定向命中率预估、NEW标识时效、DeepLink有效性 |
| 搜索栏 | 热词轮播间隔、热词过期兜底、关键词跳转行为 |

#### 阶段五：测试验证（Day 3）

| 步骤 | 操作人 | 操作内容 | 时长 |
|------|--------|----------|------|
| 5.1 | QA测试 | 在测试环境验证配置渲染效果 | 1h |
| 5.2 | QA测试 | 验证人群定向：使用不同人群包测试账号验证命中/未命中 | 1h |
| 5.3 | QA测试 | 验证展示逻辑：时间校验、频控、优先级、渠道 | 1h |
| 5.4 | QA测试 | 验证异常场景：图片加载失败、链接无效、网络超时 | 1h |
| 5.5 | QA测试 | 验证多语言文案（如有涉及） | 0.5h |
| 5.6 | QA测试 | 验证与其他资源位的互斥/层级关系 | 0.5h |
| 5.7 | QA测试 | 记录测试结果（通过/不通过/有风险） | — |

**测试用例必须覆盖**：

```
正向用例：
  ✓ 所有字段均正常填写时，配置正确渲染
  ✓ 人群定向命中用户正常展示
  ✓ 指定时间范围内正常展示
  ✓ 优先级正确判断（多条配置时取最高优先级展示）

反向用例：
  ✓ is_active = false 时不展示
  ✓ 不在投放时间范围内不展示
  ✓ 人群定向未命中用户在灰度比例内不展示
  ✓ 超出频控上限不再展示

边界用例：
  ✓ 投放时间边界（精确到分钟）正确切换
  ✓ 灰度比例 0% 和 100% 正确
  ✓ 优先级数值相等时按创建时间倒序展示
  ✓ 热词列表全部过期时回退默认文案

异常用例：
  ✓ 图片 CDN 不可用时展示占位图
  ✓ 跳转链接无效时点击行为
  ✓ 后台接口超时时使用缓存数据
  ✓ 人群包服务异常时降级策略
  ✓ 频控服务异常时降级策略

设备/渠道用例：
  ✓ iOS / Android 两端均验证
  ✓ 指定渠道/全渠道验证
```

**测试通过标准**：
- [ ] 正向用例 100% 通过
- [ ] 反向用例 100% 通过
- [ ] 边界/异常用例关键场景通过
- [ ] 两端（iOS/Android）验证一致

#### 阶段六：灰度发布（Day 4）

| 步骤 | 操作人 | 操作内容 | 观察指标 |
|------|--------|----------|----------|
| 6.1 | 配置管理员 | 设置灰度比例 = 10%，上线开关开启，**仅 A 渠道** | — |
| 6.2 | 配置管理员 | 等待 2 小时，观察数据 | 曝光量、CTR、错误率 |
| 6.3 | 配置管理员 | 数据正常 → 灰度升至 30%，扩展至 A+B 渠道 | CTR 变化趋势 |
| 6.4 | 配置管理员 | 等待 4 小时，观察数据 | CTR、CVR、异常告警 |
| 6.5 | 配置管理员 | 数据正常 → 灰度升至 50%，覆盖全部渠道 | — |
| 6.6 | 配置管理员 | 等待至次日 | 日均数据对比 |
| 6.7 | 配置管理员 | 灰度满 24h且数据正常 → 准备全量 | — |

**灰度观察指标与告警阈值**：

| 指标 | 正常范围 | 告警阈值 | 处理动作 |
|------|----------|----------|----------|
| 曝光量 | > 预期的 80% | < 预期的 50% | 检查配置生效状态 |
| CTR | 参考同类资源位基准 | 低于基准 50% 以上 | 检查图片/文案/跳转 |
| 客户端错误率 | < 0.1% | > 1% | 立即排查、准备回滚 |
| 页面加载耗时 | 无显著增加（< +100ms） | 增加 > +500ms | 检查图片大小、接口响应 |
| 告警触发 | 0 条 | ≥ 1 条 | 按告警级别处理 |

**V1.1 新增：灰度自动达标检测**

| 检测项 | 触发时机 | 处理方式 |
|--------|----------|----------|
| 灰度数据自动校验 | 每阶段灰度观察期结束时 | 自动对比当前指标与基线，全部达标则**自动推荐进入下一阶段** |
| 灰度异常自动报警 | 灰度期间任意指标触发告警阈值 | 自动暂停灰度放量，通知发布负责人评估 |
| 人群定向命中率校验 | 灰度期间每小时 | 命中率偏离预期 ±30% 时自动告警 |
| 放量建议 | 灰度满 24h 且数据全部达标 | 自动生成全量发布建议，一键审批 |

**灰度期间禁止操作**：
- ❌ 修改已灰度中配置的字段值（除非紧急）
- ❌ 调整灰度策略之外的运营控制项
- ❌ 关闭已生效的上线开关（除非触发回滚）
- ❌ 修改排期时间（会导致客户端缓存不一致）

## 四、紧急热修复流程

### 4.1 触发条件

以下情况触发紧急热修复流程：

| 场景 | 严重程度 | 响应时效 |
|------|----------|----------|
| 配置错误导致客户端 Crash | 极高 | 15 分钟内 |
| 资源位展示异常影响核心业务（如交易入口不可用） | 高 | 30 分钟内 |
| 错误内容已全量推送（PUSH/弹窗文案错误） | 高 | 30 分钟内 |
| 灰度阶段发现严重数据异常 | 中 | 1 小时内 |
| 多语言配置错误导致大面积用户投诉 | 中 | 2 小时内 |
| 人群定向配置错误（展示给错误人群） | 中 | 2 小时内 |

### 4.2 热修复流程

```
发现问题 → 快速定责 → 紧急下线 → 定位根因 → 方案确认 → 配置修正 → 验证上线 → 复盘
```

| 步骤 | 操作人 | 操作内容 | 时长 |
|------|--------|----------|------|
| 1. 发现问题 | 任何人 | 通过告警/用户反馈/测试发现异常 | — |
| 2. 快速定责 | 技术负责人 | 确认问题归属（配置/代码/数据） | 5分钟 |
| 3. 紧急下线 | 配置管理员 | 立即关闭对应资源位的 `is_active` 开关 | 1分钟 |
| 4. 定位根因 | 技术负责人 + 配置管理员 | 排查日志，定位根本原因 | 15-30分钟 |
| 5. 方案确认 | 技术负责人 | 口头确认修复方案 | 5分钟 |
| 6. 配置修正 | 配置管理员 | 修正错误配置，测试环境验证 | 15-30分钟 |
| 7. 验证上线 | 配置管理员 + 发布负责人 | 开启 `is_active`，观察数据 | 5分钟 |
| 8. 复盘 | 全团队 | 记录故障原因，更新预防措施 | 24h内 |

### 4.3 热修复审批简化规则

| 配置变更类型 | 审批要求 |
|-------------|----------|
| 仅关闭上线开关 | 无需审批，配置管理员直接操作 |
| 修正错误字段值（文案/链接/时间） | 技术负责人 + 配置管理员双人确认 |
| 调整灰度比例/频控参数 | 技术负责人确认 |
| 修正人群定向配置 | 技术负责人 + 产品运营双人确认 |
| 回退至上一版本配置 | 技术负责人 + 发布负责人双人确认 |

## 五、回滚流程

### 5.1 回滚触发条件

| 条件 | 说明 |
|------|------|
| 全量发布后 2 小时内，CTR 较灰度期间下降 > 50% | 配置可能不达预期 |
| 全量发布后错误率 > 1% 且持续超过 5 分钟 | 可能存在配置兼容性问题 |
| 全量发布后收到 ≥ 3 条严重用户投诉 | 用户体验严重受损 |
| 核心资源位（弹窗/Banner/任务）发布后数据采集异常 | 可能影响核心转化链路 |

### 5.2 回滚操作

```
确认回滚 → 关闭上线开关 → 恢复上一版本配置 → 验证 → 记录
```

| 步骤 | 操作人 | 操作内容 |
|------|--------|----------|
| 1 | 发布负责人 | 确认回滚决策，通知相关方（运营/技术/客服） |
| 2 | 配置管理员 | 关闭异常版本的 `is_active` 开关 |
| 3 | 配置管理员 | 开启上一版本的 `is_active`（如上一版本仍有效） |
| 4 | 配置管理员 | 如无可用回退版本，启用默认配置（兜底方案） |
| 5 | 配置管理员 | 验证回退后客户端渲染正常（等待缓存刷新） |
| 6 | 全团队 | 24h 内提交回滚复盘报告 |

### 5.3 兜底配置

| 资源位 | 兜底方案 |
|--------|----------|
| 弹窗运营位 | 不展示任何弹窗 |
| 顶部运营头图 | 展示默认品牌头图 |
| 底部运营四宫图 | 区域隐藏 |
| 首页跑马灯 | 区域隐藏 |
| 任务系列 | 展示默认任务模板 |
| 搜索热词 | 展示默认占位文案 |
| PUSH 推送 | 停止所有非系统级推送 |
| 导航菜单 | 展示默认菜单配置（无定向/无灰度） |

  [ ] AB 测试分组/实验ID 已关联（如适用）

## 九、附录

### 9.1 常见发布问题与预防

| 常见问题 | 原因 | 预防措施 |
|----------|------|----------|
| 上线后配置未生效 | 客户端缓存未刷新 | 等待最长缓存周期（5 分钟）后再确认 |
| 人群定向不生效 | 人群包数据 T+1 更新 | 确认人群包数据已更新后再发布 |
| 图片显示异常 | CDN 未同步 / 尺寸超标 | 发布前在测试环境验证图片加载 |
| 优先级冲突 | 同时段多配置时间重叠 | 发布前检查排期视图 |
| 灰度数据和全量数据差异大 | 灰度样本量不足 | 确保最低灰度用户数 ≥ 10,000 |
| PUSH 推送延迟 | 分批投递耗时 | 开启 batch_delivery 时预留足够发送窗口 |
| 频控配置错误 | 频控参数设置不合理 | 频控参数上线前需两人交叉验证 |
| 多语言遗漏 | 新增字段未同步所有语言包 | 发布 Checklist 中确认多语言配置 |
